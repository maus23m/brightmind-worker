| CR-019 | Feed rejection reasons back into top-up regeneration. The compute engine already writes a free-text `_computeReason` on every rejected question (e.g. "verified answer 40 not in options", "op closest_to: two candidates tied", "duplicate options"), but it is only logged to console — the top-up `generate()` call regenerates blind, so the model can repeat the same mistake. Change: collect `_computeReason` (and audit/child rejection reasons) from this job's rejected questions and inject them into the top-up prompt as a "questions rejected for these reasons — do not repeat" block. Lets Claude self-correct rather than rejection→top-up→same-error looping, and reduces wasted calls against the 26s budget. `index.js` top-up path only; pairs with a question_gen.txt note that a rejection-feedback block may be appended. Own Tester pass. | MEDIUM | ✅ DONE (26 May 2026) — `index.js`: pipeline rejections (compute/audit/child) collected with free-text `note` per question and passed into the top-up `generate()` call via the existing `rejections` param; `buildPrompt` now surfaces verbatim notes in a FEEDBACK block. Add an optional free-text textarea to the rejection modal (required when "Other" is picked, optional otherwise); store it in `question_rejections` alongside the preset `reason`. SECOND, and more important: `replaceFlagged` currently calls the Edge Function with only `count: n` — the rejection reasons are saved to the DB but never sent to regeneration, so the replacement question can repeat the rejected fault. This is the parent-facing twin of CR-019 (which feeds the compute engine's own pre-parent rejection reasons into top-up); both feedback loops are wanted.
